XMPP(6):XMPP-消息存储与拉取历史消息

post ime : 2019-04-10 category : XMPP read :

XEP-0313定义了XMPP消息存储的规则。

场景需求

0313协议主要有这些场景：

  • 同账号多客户端之间的历史消息同步
  • 客户端拉取历史消息，按日期排序展示（想想我们在微信的历史消息）
  • 分页拉取消息

存储

  1. 单条消息存储包括：
  • 消息发送跟接收的时间戳
  • from 跟 to 的JID
  • server-assigned UID
  • message stanza
  1. 消息的顺序要保存： 依赖timestamp要小心，因为多条消息可能共享时间戳
  2. 超过一定数量，可删除旧信息
  3. 群聊记录用MAM服务
  4. archive id <stanza-id/>
    被archived过的消息，server要给它加上stanza-id
    Example 1. Client receives a message that has been archived

stanza-id: archive ID

查询

1. A user queries their archive for messages

用消息UID查询

‘urn:xmpp:mam:2’ namespace, indicating the UID of the first and last message of the (possibly limited) result set.

2. Their server sends the matching messages

3. Server returns the result IQ to signal the end

server的这条iq stanza标记查询结果结束。

过滤器

1. 根据JID过滤

with 字段 + JID(Bare JID)： 会拿到to或from地址匹配JID的信息; 如果没有with, 服务端返回query指定的时间段内的消息。

使用场景：
A想查询跟B的聊天记录，with字段的value设为B, 服务端返回的messages中，既有B发送给A的msg，也有A发送给B的msg。

2. 根据接收时间过滤

start 跟 end 字段标记时间戳。 时间戳格式见https://xmpp.org/extensions/xep-0082.html

如果end 缺失， server会自动认为是最近的消息的存储时间

3. 限定results的数量

Result Set Management (XEP-0059)

这个请求，指定客户端最多只能收到10条stanzas。但服务端的返回结果可能回改变set的内容，返回自己限定的数量，比如： 这是返回start时间跟end时间段内的20条消息。

4. 分页拉取消息

如果之前已经获取了m条消息，客户端可以再发送同样的请求，拉取下一页消息。set中要带上after(上次拉取到的最后一条消息的UID)

server返回最后一页消息，会在 fin里头带上complete属性，值为ture

使用场景：

A客户端本地存储跟B的聊天信息， 最后一条message的id是09af3-cc343-b409f。 现在A想看看最近的消息（09af3-cc343-b409f后的message，可以发送iq请求中带上<after>09af3-cc343-b409f</after>。差量请求最新消息，基于游标的分页。

5.其他字段的筛选

客户端查询服务端支持的其他字段

返回的message stanza 结构

  • message被封装在forwarded元素中。 XEP-0297: Stanza Forwarding
  • 带result 元素， 其属性id是这条message的UID
  • delay元素 XEP-0203: Delayed Delivery message被收到的时间, UTC时间戳格式

MUC Archive

  • 存储所有发送给roomJid的message
  • 不包含private message
  • user需要权限查询群历史聊天记录
  • forward stanza中带有to属性,值是roomJid，from值是userJid
  • x里有该消息的发送者Jid
